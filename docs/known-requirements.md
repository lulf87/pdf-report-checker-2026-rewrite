## 1. 产品定位

### 1.1 工具名称

- 中文名：报告核对工具。
- 英文/产品名：Report Checker Pro。

### 1.2 使用对象

- 医疗器械检验报告核对人员。
- 报告编制、复核、质量管理或技术审评支持人员。

### 1.3 核心目标

- 自动核对检验报告与 PTR 产品技术要求之间的条款摘录一致性。
- 自动核对检验报告内部一致性，包括首页/第三页字段、样品描述、照片、中文标签、检验项目结论、非空字段、序号、续表和页码。
- 输出可追溯证据，帮助人工优先处理明确错误和需复核项；系统不替代最终人工签发。

## 4. 功能模块一：PTR 条款核对

### 4.1 核对目标与范围

- PTR 第 2 章为核对目标。
- 章节名称不固定，应以章节编号 `2` 及其子编号定位，不应依赖固定标题。
- TODO：旧 2026.4.13 实现中 `_ptr_chapter_2_text_pages` 对 `2 性能指标` 有固定标题依赖迹象；重做时需按“编号定位”修正。
- 报告侧以首页/第三页“检验项目”声明的条款范围为本轮核对范围；报告首页未声明的 PTR 第 2 章条款不应作为缺失项。
- 首页明确排除的项目，如生物相容性、电磁兼容性等，不纳入本轮缺失或不一致。

### 4.2 PTR 条款层级解析

- 支持 `2`、`2.1`、`2.1.1`、`2.1.1.1` 等层级。
- 支持首页检验项目中的 exact selector，例如 `2.1.5`。
- 支持范围 selector，例如 `2.1～2.4`、`2.1.2～2.1.4`。
- 对父条款范围，应展开其下被声明范围内的最细叶子条款。
- 对只有分组作用、无主要求正文的父条款，应作为上下文或分组处理，是否展示为 INFO/TODO 需确认。

### 4.3 非主要求内容排除

以下内容不进入 PTR 主要求严格一致性统计：

- 检验方法、试验方法、测试方法、检测方法章节。
- 附录、说明、注释、图、示意图、纯解释性文本。
- 外部标准全文或外部标准条款，例如 GB、GB/T、YY、YY/T 等引用标准的完整内容。
- 报告中“标准的内容”区段或非检验项目表“标准要求”列的说明性范围，不作为 PTR 主比对源。

### 4.4 条款正文比对

- 条款正文要求严格匹配。
- `≥/≤` 与 `>/<` 差异按不一致处理，不能视为等价表达。
- 报告可以展开 PTR 引用表格；只要标准要求完整覆盖对应 PTR 条款即可。
- 若报告使用“见序号 1～序号 118”、GB 标准号或附录索引概括安全要求，且核心要求已覆盖，不应仅因索引式摘要判定为问题。旧 `PTR-2.4` 测试和服务中有专门的 warning suppression。

### 4.5 比对前标准化

允许用于比对前的标准化：

- 全角/半角统一。
- 自然换行合并。
- 多余空格折叠或去除。
- OCR 提取导致的无意义断行修复。

### 4.6 `见表 X` 展开表格比对

- PTR 条款正文出现“见表 X”“符合表 X”“按表 X”等引用时，必须解析并展开对应表格。
- 表格比对应覆盖表号、表题、参数名、参数值、单位、适用型号/条件和允许误差。
- 表格参数名和值均需比对；不能只确认表号出现。
- 多页表格、续表、重复表号候选、多行表头需保留诊断信息；无法可靠拼接或选择候选时应 WARN。
- 表格比对的结果应关联回引用该表的 PTR 条款。

### 4.7 结果展示

- Web 结果必须支持 PTR 内容 / 报告内容对照。
- 必须支持差异高亮，包括文本差异和表格参数差异。
- PTR 结果默认或至少提供“仅显示不一致项”视图；通过项可保留在 JSON 中用于审计，但 UI 的问题处理视图应只显示不一致/需复核项。
- 缺漏条款、额外条款、不一致条款、证据不足条款需要分级展示。

## 6. OCR 与 Caption 规则

### 6.1 Caption 定义

- Caption 指照片页或照片说明区中与图片/标签对应的标题行或说明行。
- 旧项目文本样例包括 `№7 导管标签样张`、`№6 消融仪发生器 中文标签样张`、`检品外观` 等。
- 照片页识别关键词：`检验报告照片页`、`照片和说明`。
- 标签页识别关键词：照片页文本中包含 `标签`，尤其是 `中文标签`、`中文标签样张`、`标签样张`。

### 6.2 主体名提取规则

主体名用于把样品描述部件与照片/标签 caption 匹配：

- 去除编号：`№7`、`No.7`、`编号7`、开头数字序号。
- 去除类别词：`照片`、`图片`、`外观`、`检品外观`、`标签`、`中文标签`、`标签样张`、`包装标签`。
- 去除方位词或拍摄角度词：如 `正面`、`背面`、`侧面`、`局部`、`整体`。
- 去除多余空格、自然换行、全半角差异。
- 保留能区分部件的核心名称、型号、批号/序列号等信息。

### 6.3 中文标签 OCR 字段正则

旧实现 `_extract_label_fields` 使用字段名列表加冒号分隔抽取。基线字段包括：

- 产品名称、样品名称、部件名称。
- 型号规格、规格型号、型号/规格、型号。
- 产品编号/批号、批号/序列号、序列号/批号、批号、序列号。
- 生产日期、失效日期。

字段抽取模式：

- 支持中文冒号和英文冒号：`字段名[:：]字段值`。
- 字段值到下一个字段名或文本结尾为止。
- 同一页可抽取多个字段。

生产日期正则基线：

- 至少支持 `YYYY-MM-DD`、`YYYY/MM/DD`、`YYYY.MM.DD`、`YYYYMMDD`。
- 旧代码日期外观判断接近：`\d{4}[-/.年]?\d{1,2}[-/.月]?\d{0,2}日?`。

批号、序列号、失效日期字段：

- 失效日期需支持 `失效日期`、`有效期至`、`EXP`。TODO：旧代码仅明确有 `失效日期` 字段名，其他别名需确认。

### 6.4 OCR / LLM / VLM 判定边界

- OCR 置信度低时，相关字段或规则应输出 WARN，不得静默通过。
- 普通 LLM/VLM adapter 只能用于增强提取、图片文字审阅或候选匹配说明。
- Codex CLI 是单独的产品运行时 auditor / judge，可在受控 sandbox、timeout 和 JSON schema 下复核 candidate findings。
- 旧 2026.4.13 实现使用 Codex 作为多个检查项的 judge；重做项目不再采用 evidence-only 假设，而是拆分为“确定性规则初判”和“Codex 运行时审核”两层证据。
- Codex review 可以 `confirm`、`refute`、`uncertain`、`add_finding`，但不得静默删除原始 `Finding`。
- OCR/VLM 失败不应导致整个主流程崩溃；只有必需数据无法获得且无法降级时，相关任务或规则才进入 error/WARN 状态。

### 7.3 任务状态

本基线要求任务状态：

- `pending`：已创建，等待执行。
- `processing`：正在解析、OCR、核对或导出。
- `completed`：处理完成，结果可获取。
- `error`：任务失败，需返回可读错误。

旧 2026.4.13 实际任务状态主要为：

- `running`
- `completed`
- `error`

如果新架构改为 `queued / running / succeeded / failed`，映射关系如下：

| 基线状态 | 新状态可选映射 | 旧实现近似状态 |
|---|---|---|
| `pending` | `queued` | 旧实现创建后很快进入 `running`，缺少稳定 pending。 |
| `processing` | `running` | `running`。 |
| `completed` | `succeeded` | `completed`。 |
| `error` | `failed` | `error`。 |

### 7.4 进度与错误

- 任务返回 `task_id`、文件名、当前检查项、已完成数量、总数量、日志、结果和错误信息。
- 轮询间隔旧前端为 2 秒。
- 解析失败、非 PDF、损坏 PDF、Codex/OCR 调用失败应有用户可读错误；不能只返回堆栈。
- 对单条规则的解析失败应尽量降级为 WARN/missing evidence，避免整份报告失败；但 PDF 无法打开、必需文件缺失等基础错误可导致任务 error。
