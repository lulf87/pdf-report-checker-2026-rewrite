# AGENTS.md

本文件是本仓库的长期 Codex 工作规则。除非子目录中存在更近的 `AGENTS.md` 并给出更具体规则，否则 Codex 在本项目中的所有开发、重构、测试、核对、文档和交付工作都必须遵守本文件。

## 项目目标

- 本项目用于医疗器械检验报告自动核对。
- 本项目不是从空白开始，而是基于旧项目资产重构。
- 必须保留旧项目已有业务能力，并重做架构。
- 旧项目业务能力至少包括：
  - PTR 条款核对。
  - 报告自身核对 C01-C11。
- 技术栈保持为：
  - 后端：FastAPI。
  - 前端：React + TypeScript + Vite。
  - PDF：PyMuPDF。
  - OCR：PaddleOCR。
- 新架构必须可测试、可维护、规则独立。
- 优先做最小有效变更，不做无关重构，不凭空扩展业务规则。

## 旧项目资产原则

- `README.md`、`REPORT_CHECKER_SPEC.md`、`CLAUDE.md`、`backend/tests` 是业务依据。
- 旧 `services` 中的算法经验可以迁移，但迁移时必须放入新架构的合适层级。
- 旧 `router` 结构不直接迁移。
- 旧 Electron、`python_backend`、`src/renderer` 不进入新主线。
- `uploads`、`temp`、`logs` 不作为源码资产。
- 不要直接复制旧混乱架构。
- 不要删除旧代码，除非任务明确要求。
- 如果旧资产与新架构规则冲突，保留业务语义，重做实现边界。
- 迁移旧逻辑时，必须能追溯到旧文件、测试、规格文档或用户明确指令。

## 后端架构规则

- 后端分层职责必须清晰：
  - `api` 层只处理 HTTP 请求/响应、参数解析、状态码和序列化。
  - `application` 层负责编排 usecase、事务边界、任务流程和跨服务调用。
  - `domain` 层定义稳定业务模型、业务概念、枚举和值对象。
  - `rules` 层实现 C01-C11 和 PTR 规则。
  - `infrastructure` 层处理 PDF、OCR、LLM、文件、导出、外部系统适配。
- 不允许在 router 中写业务规则。
- 不允许在 PDF parser 中写核对规则。
- 不允许在 OCR、文件解析、导出适配器中混入 C01-C11 或 PTR 判断。
- 不允许用临时 `dict` 作为跨层契约，优先使用 Pydantic model。
- 所有规则必须输出统一 `Finding`。
- `Finding` 的字段、严重级别、定位信息和证据结构必须由稳定模型定义，不能由各规则自由拼接。
- 规则输入和输出必须适合单元测试，避免直接依赖 HTTP、文件系统、前端状态或全局变量。
- 后端新增能力时，优先补齐 model、usecase、rule、infrastructure adapter 的边界，再接入 API。
- 如需接入 LLM，LLM 只能作为 infrastructure 能力或辅助解释能力，不得替代确定性核对规则。

## 报告自身核对规则

必须保留 C01-C11，编号和语义不得随意改名或合并：

- C01：首页与第三页字段一致性。
- C02：第三页扩展字段与中文标签 OCR。
- C03：生产日期格式和值一致性。
- C04：样品描述表格与中文标签 OCR。
- C05：照片覆盖。
- C06：中文标签覆盖。
- C07：单项结论逻辑。
- C08：非空字段。
- C09：序号连续性。
- C10：续表标记。
- C11：页码连续性。

实现 C01-C11 时必须遵守：

- 每个规则保持独立，可单独启用、单独测试、单独定位失败原因。
- 不要在前端组件中实现 C01-C11 判断。
- 不要在 PDF parser 或 OCR parser 中直接返回最终核对结论。
- 字段提取、OCR 识别、表格解析、规则判断必须保持边界清晰。
- 不确定的字段映射、格式解释、业务例外必须标记为待确认，不得自行拍脑袋定死。

## PTR 核对规则

- PTR 第 2 章为主核对范围。
- 条款层级必须保留。
- 检验方法、附录、说明、图等不进入主一致性统计。
- 条款正文严格匹配。
- `见表 X` 必须展开表格参数比对。
- 标准条款范围排除逻辑必须保留。
- PTR extractor 必须保留章节、条款、表格、排除范围、正文证据之间的可追溯关系。
- 条款匹配失败时，应输出可定位的 `Finding`，包含来源、目标、差异和证据。
- 不要把 PTR 主一致性统计与辅助说明、附录、图示、检验方法混在同一口径中。

## 测试规则

- 每个规则必须有独立 pytest。
- 每个 usecase 必须有集成测试。
- API 必须有测试。
- PDF parser、OCR、PTR extractor、table normalizer 必须有测试。
- Golden File 测试是核心保障。
- 不要修改 Golden expected，除非任务明确要求并说明原因。
- 后端任务完成后运行：

```bash
cd backend && pytest tests/ -v
```

- 前端任务完成后运行：

```bash
cd frontend && npm run build
```

- 如果项目中接入 Playwright，则前端联调任务必须浏览器实测。
- 不要跳过测试。若测试因环境、依赖、样本文件或外部服务无法运行，必须在最终报告中明确说明原因和未验证风险。
- 不得声称规则、API、构建或联调已完成，除非已经实际运行对应验证。

## 前端规则

- 前端只展示后端结果，不实现 C01-C11 判断。
- 类型定义必须与后端 API 对齐。
- 使用 React + TypeScript + Vite。
- UI 保留旧项目深色玻璃拟态方向。
- TailwindCSS v4 使用 CSS-first。
- Framer Motion 动效参数集中管理。
- 不要在组件中硬编码复杂业务判断。
- 前端可以做筛选、排序、分组、展开、折叠、状态展示和用户交互，但不得重新计算核对结论。
- API response type、Finding type、任务状态 type 应保持集中定义，避免组件内散落临时类型。
- 前端展示差异、证据、页码、字段名、规则编号时，必须优先使用后端返回的结构化字段。

## 数据、文件和生成物规则

- 不要修改原始输入数据，除非任务明确要求进行 raw-data repair，且目标文件已复制到专用修复工作区。
- 不要提交或纳入源码主线：
  - `uploads`
  - `temp`
  - `logs`
  - `node_modules`
  - `.venv`
  - `dist`
  - `__pycache__`
- 生成文件、缓存文件、上传样本、临时日志不得作为源码资产。
- 不要静默覆盖重要内容。
- 不要重命名高层目录，除非任务明确要求。
- 新增配置、样例、fixture 或 Golden File 时，必须说明用途和来源。

## 业务判断和证据规则

- 不要伪造引用、数据集、实验结果、统计值、参数值、文件路径或执行结果。
- 不要自行编造医疗器械检验规则、标准条款、字段含义或例外条件。
- 每个实质性业务判断应能追溯到：
  - 项目规格文档。
  - 旧项目代码或测试。
  - Golden File。
  - 用户明确指令。
  - 可复现的解析输出或核对输出。
- 如果规则存在歧义，应输出待确认项，而不是把不确定逻辑固化为代码。
- 如果结果无法在当前环境复现，不得报告为已验证。

## 开发和迁移规则

- 保留旧项目业务能力，重做架构边界。
- 从旧代码迁移算法时，先识别输入、输出、规则意图和测试依据，再放入新架构。
- 优先迁移业务价值明确、测试可覆盖的逻辑。
- 不要为了复用旧代码而牺牲新架构分层。
- 不要为了短期跑通而把业务规则塞进 API、parser、OCR adapter 或前端组件。
- 新增依赖必须必要，并说明理由。
- 新增抽象必须解决真实复杂度，不能为了形式化而增加层级。
- 修复缺陷时优先修 root cause，避免只 suppress error。

## 完成标准

每个任务完成后必须输出：

1. 修改文件清单。
2. 实现说明。
3. 测试命令。
4. 测试结果。
5. 未完成项。
6. 风险或需要人工确认的问题。

最终报告必须清楚区分：

- 已修改内容。
- 已运行验证。
- 未运行验证。
- 剩余风险。
- 需要用户确认的业务问题。
